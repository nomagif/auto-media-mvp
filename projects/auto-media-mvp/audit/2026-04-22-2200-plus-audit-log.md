## Cause memo

Observed pattern:
1. A user message enters the session.
2. The assistant starts work and may emit tool calls.
3. A custom internal failure event (`openclaw:prompt-error`) records an idle timeout.
4. The system snapshots / switches provider state and replays the same user payload.
5. The fallback attempt fails with provider quota exhaustion.
6. Because no final assistant text message is completed, simplified history views can look like conversation items “disappeared.”

Likely explanation:
- The messages are not primarily lost from the raw transcript.
- Instead, the user-facing / simplified history view omits or compresses internal custom events.
- When a run dies between tool activity and final answer, the raw JSONL preserves the failure chain, but the simplified history can appear to skip or collapse those steps.

Practical implication:
- For incident review, use the JSONL transcript as the source of truth.
- For operator-facing summaries, present three buckets separately:
  - successful outcomes,
  - failed outcomes,
  - internal hidden errors.
